# Live Evidence Proof

This document explains how Decoda RWA Guard validates and proves live operational evidence
as required by NIST CSF 2.0's DETECT → RESPOND chain.

## What counts as live evidence

Live evidence must satisfy all of the following:

1. **Provider configured**: `EVM_RPC_URL` is set, non-placeholder, and resolves to a live
   chain endpoint. `EVM_CHAIN_ID` is recommended for explicit chain identification.

2. **Live telemetry received**: `last_telemetry_at` must be populated. This proves that
   monitored chain data actually arrived from the live provider.

3. **Detection generated from telemetry**: At least one detection must exist and be linked
   to the telemetry event by ID or evidence lineage.

4. **Alert generated from detection**: At least one alert must be linked to the detection.

5. **Incident or response-action linked to alert**: At least one incident or response action
   must be linked to the alert, closing the DETECT → RESPOND loop.

6. **Exportable evidence bundle**: A proof bundle must be exportable and truthfully labeled
   as `live` (not `simulator`, `demo`, or `unknown`).

## Why heartbeat and poll are NOT telemetry

These are three distinct signals:

| Signal | What it proves | What it does NOT prove |
|---|---|---|
| **Heartbeat** | The monitoring worker process is alive | Monitored chain data arrived |
| **Poll** | The monitoring loop executed (iterated) | Monitored chain data arrived |
| **Telemetry** | Monitored chain data was received and stored | Nothing else |

A system can have a healthy heartbeat and regular polls with **zero telemetry**. This means
the worker is running but no chain data has been ingested. In that state:

- `live_evidence_chain_ready = false`
- Broad SaaS readiness is **blocked**

The `check_live_evidence_chain()` function enforces this: if `last_telemetry_at` is absent,
it emits a specific blocker regardless of heartbeat/poll status.

## Evidence source labels

| Label | Meaning | Satisfies live evidence? |
|---|---|---|
| `live` / `live_provider` | Data from real chain provider | Yes |
| `simulator` / `guided_simulator` | Generated by local simulation | **No** |
| `demo` | Demo/fixture data | **No** |
| `fixture` | Hard-coded test fixture | **No** |
| `unknown` | Source not declared | **No** (fail-closed) |

## Contradiction flags that invalidate live evidence

If the runtime monitoring summary contains any of these contradiction flags, live evidence
is blocked even if the source is labeled `live`:

- `live_mode_with_simulator_evidence`
- `simulator_evidence_present`
- `healthy_without_telemetry`
- `live_mode_without_telemetry`
- `missing_telemetry`

These flags are detected by `detect_runtime_contradictions()` in
`services/api/app/runtime_truthfulness.py`.

## How to configure a live EVM provider

1. Set `EVM_RPC_URL` to a real Ethereum mainnet (or target network) endpoint:
   - Infura: `https://mainnet.infura.io/v3/<project-id>`
   - Alchemy: `https://eth-mainnet.g.alchemy.com/v2/<api-key>`
   - Self-hosted node: `https://your-node-url:8545`

2. Set `EVM_CHAIN_ID` to the target chain integer ID (e.g., `1` for Ethereum mainnet).

3. Verify provider health returns a current block number.

## Running chain validation

```python
from services.api.app.paid_launch_readiness import check_live_evidence_chain

result = check_live_evidence_chain({
    'evidence_source': 'live',
    'last_heartbeat_at': '2026-01-01T00:00:00Z',
    'latest_poll_at': '2026-01-01T00:00:30Z',
    'last_telemetry_at': '2026-01-01T00:01:00Z',
    'detections_count': 1,
    'detection_telemetry_linked': True,
    'alerts_count': 1,
    'alert_detection_linked': True,
    'incidents_count': 1,
    'incident_alert_linked': True,
    'export_capability': 'pass',
    'export_source_label': 'live',
    'contradiction_flags': [],
})

print(result['live_evidence_chain_ready'])  # True
print(result['chain_blockers'])             # []
```

## NIST CSF 2.0 alignment

| NIST function | Decoda evidence |
|---|---|
| **GOVERN** | Workspace isolation, RBAC, audit log |
| **IDENTIFY** | Asset registry, target configuration, monitoring config |
| **PROTECT** | Auth/session, API key management, MFA |
| **DETECT** | Live telemetry → detection pipeline |
| **RESPOND** | Alert → incident → response-action chain |
| **RECOVER** | Evidence export, proof bundle, audit trail |

The Detect → Respond chain is proven by `live_evidence_chain_ready=true` in the launch proof.

## What output is required before claiming safe to sell broadly today

The following must all be true in `artifacts/final-readiness/latest/summary.json`:

```json
{
  "controlled_pilot_ready": true,
  "broad_paid_saas_ready": true,
  "production_100_percent_ready": true,
  "safe_to_sell_broadly_today": true,
  "required_gates": {
    "live_evidence_readiness": { "status": "pass" }
  }
}
```

And the launch proof must show:

```json
{
  "readiness": {
    "live_evidence_ready": true,
    "provider_ready": true
  }
}
```

If `safe_to_sell_broadly_today` is `false`, the `safe_to_sell_reason` field explains why.

## Simulator evidence policy

Simulator/demo/fixture evidence:
- May be used for **controlled pilot** demonstrations.
- Must be clearly labeled in all exports as `simulator`, never as `live`.
- **Does not** satisfy `live_evidence_ready` for broad SaaS readiness.
- **Does not** satisfy NIST DETECT function evidence requirements for production claims.

The `_live_provider_proof_present()` function and `check_live_evidence_chain()` both enforce
this rule. There is no flag, environment variable, or workaround that makes simulator
evidence count as live evidence for `broad_paid_saas_ready`.
